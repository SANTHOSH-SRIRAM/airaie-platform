# Plan 09-02 — Heavy renderer + governance completeness (Phase 2b–2f closure)

## Goal

Close the remaining Phase 2b–2f scope from `concepts/04-RENDERER-REGISTRY.md`. Deliver 6 specific outcomes across 4 sub-waves (~5 dev-days total). Each item is independently shippable as an atomic commit; the wave grouping is for context, not enforcement.

After this plan, the renderer registry is feature-complete per the original Phase 9 design contract. The only deferred items will be net-new design work surfaced during execution (e.g. annotation overlays, multi-board comparison) rather than gaps in the original spec.

## Success criteria (what must be TRUE on completion)

1. User opens a Card with an image artifact (PNG/SVG/JPG) and can scroll-wheel zoom or pinch-zoom on it; pan via drag works inside the zoomed viewport
2. User opens a Card from a CalculiX FEA run and sees the deformed surface mesh + von Mises field rendered inline by `VtpViewer` (not a download link); the underlying `.frd` is accessible alongside the inline view
3. A tool's ATP manifest declaring `output.renderer_hint = "csv-chart"` results in the chart renderer mounting (via `Artifact.metadata.renderer_hint`) even when the kind-only fallback would pick `csv-table`
4. A `comparison` Card renders two FEA result decks side-by-side via `SplitRenderer axisLocked`, and rotating one viewer rotates the other
5. A running CFD job's residuals appear live in a small Recharts line chart that updates per SSE event (no full-page poll), gated behind a per-tool `streaming_chart` manifest hint
6. A Run completed in Release mode locks the camera position + scalar range when re-opened in any future session; Explore-mode reads remain free

### 2E.1 Streaming residuals chart (~1-2 days)

**Problem.** CFD/FEA tools emit residuals (per-iteration scalars) during a Run. Currently a user has to wait for the run to complete and open the result deck to see convergence. We want a live mini-chart in the active stage as the run streams.

**Files**
- `frontend/src/renderers/StreamingResidualsRenderer.tsx` (new) — Recharts line chart that subscribes to a per-run SSE channel; appends points as events arrive
- `frontend/src/renderers/registry.ts` — register `streaming-residuals` entry, gated by `manifest hint: "streaming-residuals"` on the source tool (does NOT match by kind alone — this is hint-only since residuals look like generic streaming data otherwise)
- `frontend/src/hooks/useStreamingResiduals.ts` (new) — wraps the existing `useRunSSE` pattern, filters events for `event_type: "residual"`, returns `{ points: [{iter, value, label}], isStreaming }`
- `airaie-kernel/internal/runner/...` — verify the runner already pumps `event_type: "residual"` events when tools emit them. If not, define the event shape
- `atp/examples/openfoam-cavity/run_live.py` — wire residual emission (write residuals to a port the runner reads + re-emits as SSE events)
- Tests: `frontend/src/renderers/StreamingResidualsRenderer.test.tsx` — mock SSE source, push events, assert chart updates

**Action**
1. **Day 1 morning** — define the SSE event shape: `{ event_type: "residual", iteration: int, residual_name: str, value: float, ts: iso }`. Update `EventEnvelope` in runner if needed
2. **Day 1 afternoon** — `useStreamingResiduals` hook: filter `useRunSSE` events for residuals, accumulate in state, expose a sliding window of last N=500 points
3. **Day 2 morning** — `<StreamingResidualsRenderer>` Recharts component: line chart with log-Y option (CFD residuals span orders of magnitude), legend per residual name, `isStreaming` indicator pill
4. **Day 2 afternoon** — register in registry as `id: 'streaming-residuals'`, predicate matches when `artifact.metadata.renderer_hint === 'streaming-residuals'` AND the run is `running` (don't show the chart for completed runs — they have static plots)
5. Test: mock SSE, push 100 residual events, assert chart points = 100, log-Y toggle works
6. Wire OpenFOAM example to emit residuals (most CFD solvers print to stdout — parse + emit)

**Verify**
- `npx vitest run src/renderers/StreamingResidualsRenderer.test.tsx` passes

**Problem.** With `view_state` persistence in place, Release mode needs to enforce immutability: the user can pan/rotate locally for inspection, but the persisted `view_state` doesn't change, AND on first open the camera is hard-locked to the persisted state (no animated easing into a "default" view that drifts run-to-run).

**Files**

## Risks & mitigations

1. **vtk.js bundle bloat** — `render-vtk` chunk is already ~2 MB; adding shared view-state coordination doesn't grow it materially. Watch.
2. **SSE backpressure for residuals** (2E.1) — already mitigated in the action plan (250ms throttle on hook side).
3. **CalculiX image rebuild + re-register** (2B.3) — if the image bump isn't picked up by the runner, FRD→VTP silently degrades. Mitigation: live UAT is a hard gate; if `.vtp` doesn't appear in the run's outputs, the bump didn't take.
4. **`renderer_hint` hint typos** (2B.2) — frontend tier-1 lookup falls through gracefully to tier-2 if the hint references an unregistered id. Document in the ATP spec that hints are advisory.
5. **Release-mode lock subjective** (2F.2) — user may want strict camera lock (no local rotation at all) instead of the proposed "local-rotation-allowed-but-not-persisted." If the design decision drifts during execution, file a 999.x for the alternative.

## Out of scope (file in 999.x if needed)

- Annotation overlays on 3D / VTP viewers (post-it notes, dimension callouts) — not in `concepts/04`
- Multi-board comparison (3+ runs in one view) — `SplitRenderer` is two-up only
- View_state diff / merge across users (collaborative editing)
- Renderer hint marketplace / discovery (let users override per-card without changing the manifest)
- PDF rich features — annotation, page-pin (current iframe path is sufficient)
