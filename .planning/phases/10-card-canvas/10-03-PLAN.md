# Plan 10-03 — Real KPI / Method / Run NodeViews + Full Slash Menu

<objective>
Replace the shared `TypedBlockPlaceholder` for the next three typed governance blocks (`kpiBlock`, `methodBlock`, `runBlock`) with real, data-bound React NodeViews; expand the slash menu shipped in 10-02 (3 items) into a full menu surfacing all 11 typed-governance kinds plus 5 layout kinds with cardinality enforcement; and extend `cardCanvasContext` so the new Method and Run NodeViews receive `card`, `intent`, and `plan` directly from the page rather than each calling its own data hooks (n+1 fetch avoided, single source of truth).

Concretely:

- `KpiBlockView` reads `attrs.metricKey | attrs.operator | attrs.threshold` directly from the node attrs (no fetch — the attrs ARE the data per 10-RESEARCH §"The block schema"). When a `latestRun` is available via `cardCanvasContext`, KpiBlockView optionally derives the measured value from `intent.acceptance_criteria` cross-referenced against run output metrics (best-effort; falls back to `'pending'` when no measurement is reachable). Renders a status chip (`✓ pass` / `✗ fail` / `— pending`) computed by the pure `evaluateKpi` helper.

- `MethodBlockView` calls `usePlan(cardCanvasContext.cardId)` — NOT `usePlan(attrs.planId)`. The hook's signature only accepts a Card ID (verified by reading `frontend/src/hooks/usePlans.ts:30`); `attrs.planId` is preserved on the node for serialization stability across plan regenerations but is NEVER used as the fetch key. This contract mismatch with the 10-RESEARCH table (which says Method's NodeView fetches via `usePlan({ planId })`) is resolved by routing through `cardId`. Empty state when no plan exists (renders an inline "Generate plan" button calling `useGeneratePlan(cardId).mutate()`). Loaded state shows pipeline_id, step count, plan status, and a one-line list of node tool_ids.

- `RunBlockView` is the user-facing inline Run button — the canvas equivalent of `CardActionBar`. It pulls `card`, `intent`, and `plan` from `cardCanvasContext` and instantiates `useCardRunState(card, intent, plan)`. It then renders one of the 7 stage-specific UI branches (no-intent / no-inputs / no-plan / ready / running / completed / failed). The action handler each branch wires up (`state.generate()`, `state.run()`, `state.cancel()`, `state.rerun()`) is invoked directly. Visual treatment matches `CardActionBar`'s vocabulary (label + variant + icon) without re-using the floating bottom-bar chrome — this is an inline NodeView, not a fixed bar. CardActionBar continues to mount and remain functional; the two surfaces share `useCardRunState` and so flip together within one render cycle.

- Full slash menu — `getSlashMenuItems` ships 3 items in 10-02 (Intent, Input, Result). 10-03 expands it to 16: all 11 typed-governance kinds plus 5 layout kinds (Heading 1, Heading 2, Bullet List, Code Block, Divider). Cardinality rules per 10-RESEARCH §"Cardinality enforcement": `Intent`, `Method`, `Run` are each hidden when the doc already contains the corresponding block (1-per-Card). The other 8 typed kinds and all 5 layout kinds are 0..N. New cardinality helpers `docHasMethodBlock` and `docHasRunBlock` mirror the shape of 10-02's `docHasIntentBlock`. SlashMenuPopover gains a small visual section break between the typed-governance group and the layout group — purely a render change; the underlying `SlashMenuItem[]` API is unchanged.

- `cardCanvasContext` extends from `{ cardId, boardId, intentSpecId }` (10-02) to `{ cardId, boardId, intentSpecId, card, intent, plan }`. `CardCanvasPage` populates the rich fields by passing through its existing `useCard / useIntent / usePlan` query data (already fetched — no new round trips). The default value preserves graceful degradation: NodeViews mounted outside a `<CardCanvasContext.Provider>` see `card: null, intent: undefined, plan: null` and render their respective empty states.

This wave intentionally ships NO Evidence / Gate / EmbedCard / EmbedRecord / AiAssist NodeViews (they continue to mount `TypedBlockPlaceholder` until 10-04), NO drag-drop right palette (10-04), NO Mode-rule per-block edit locks (10-06), NO 3-way merge UI on 409 (10-06), NO Board canvas (10-05), and NO new backend work (kernel is fully ready).

   ```ts
   import type { KpiBlockNode } from '@/types/cardBlocks';

   export type KpiOperator = KpiBlockNode['attrs']['operator'];
   export type KpiThreshold = KpiBlockNode['attrs']['threshold'];
   export type KpiResult = 'pass' | 'fail' | 'pending';

   /**
    * Pure helper — render a one-line KPI summary. Used by KpiBlockView's
    * loaded chrome.
    *
    * Examples:
    *   { metricKey: 'drag_coefficient', operator: 'less_than', threshold: 0.3 }
    *     -> 'drag_coefficient < 0.3'
    *
    *   { metricKey: 'reynolds', operator: 'between', threshold: [1e5, 5e5] }
    *     -> 'reynolds in [100000, 500000]'
    *
    *   { metricKey: '', ... } -> ''   (empty — caller renders empty state)
    */
   export function formatKpiSummary(attrs: {
     metricKey: string;
     operator: KpiOperator;
     threshold: KpiThreshold;
   }): string {
     if (!attrs.metricKey || attrs.metricKey.trim() === '') return '';
     switch (attrs.operator) {
       case 'less_than':
         return `${attrs.metricKey} < ${formatNumber(attrs.threshold)}`;
       case 'greater_than':
         return `${attrs.metricKey} > ${formatNumber(attrs.threshold)}`;
       case 'eq':
         return `${attrs.metricKey} = ${formatNumber(attrs.threshold)}`;
       case 'between': {
         if (Array.isArray(attrs.threshold) && attrs.threshold.length === 2) {
           const [lo, hi] = attrs.threshold;
           return `${attrs.metricKey} in [${formatNumber(lo)}, ${formatNumber(hi)}]`;
         }
         return `${attrs.metricKey} in [?]`;
       }
     }
   }

   function formatNumber(n: number | [number, number]): string {
     if (Array.isArray(n)) return `${n[0]}..${n[1]}`;
     return Number.isInteger(n) ? String(n) : n.toString();
   }

   /**
    * Pure helper — evaluate a KPI's pass/fail status given a measured value
    * (from a completed run) or undefined (no measurement available yet).
    *
    *   evaluateKpi('less_than', 0.3, 0.25) -> 'pass'
    *   evaluateKpi('less_than', 0.3, 0.4)  -> 'fail'
    *   evaluateKpi('less_than', 0.3, undefined) -> 'pending'
    *   evaluateKpi('between', [10, 20], 15) -> 'pass'
    *   evaluateKpi('between', [10, 20], 25) -> 'fail'
    *   evaluateKpi('eq', 5, 5)   -> 'pass'
    *   evaluateKpi('eq', 5, 5.1) -> 'fail'
    *
    * Tested across all 4 operators + the no-measurement case.
    */
   export function evaluateKpi(
     operator: KpiOperator,
     threshold: KpiThreshold,
     measured: number | undefined,
   ): KpiResult {
     if (measured === undefined || !Number.isFinite(measured)) return 'pending';
     switch (operator) {
       case 'less_than':
         return typeof threshold === 'number' && measured < threshold ? 'pass' : 'fail';
       case 'greater_than':
         return typeof threshold === 'number' && measured > threshold ? 'pass' : 'fail';
       case 'eq':
         return typeof threshold === 'number' && measured === threshold ? 'pass' : 'fail';
       case 'between': {
         if (Array.isArray(threshold) && threshold.length === 2) {
           const [lo, hi] = threshold;
           return measured >= lo && measured <= hi ? 'pass' : 'fail';
         }
         return 'pending';
       }
     }
   }
   ```
