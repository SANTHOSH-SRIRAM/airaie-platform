## Task breakdown

### Task 1 — `getRecord` API + `useRecord` hook + `boardKeys.record`

Add to `frontend/src/api/boards.ts`:
```ts
export async function getRecord(boardId: string, recordId: string): Promise<BoardRecord> {
  return apiClient.get<BoardRecord>(`/v0/boards/${boardId}/records/${recordId}`);
}
```
(Verify response shape — kernel returns the BoardRecord directly per `boards.go:314`. If envelope-wrapped, adapt with `unwrapList` or unwrap helper.)

Add to `frontend/src/hooks/useBoards.ts`:
```ts
export const boardKeys = {
  // ... existing keys
  record: (boardId: string, recordId: string) =>
    ['boards', boardId, 'record', recordId] as const,
};

export function useRecord(boardId: string | undefined, recordId: string | undefined) {
  return useQuery({
    queryKey: boardKeys.record(boardId ?? '__missing__', recordId ?? '__missing__'),
    queryFn: () => getRecord(boardId!, recordId!),
    enabled: !!boardId && !!recordId,
    staleTime: 30_000,
  });
}
```

### Task 2 — `EvidenceBlockView` + helpers + tests

`EvidenceBlockView.helpers.ts` exports pure `findEvidenceById(list: CardEvidence[], id: string | null): CardEvidence | undefined` and `formatEvidenceSummary(ev: CardEvidence): { line1: string; line2: string; chip: 'pass' | 'fail' | 'pending' }`.

`EvidenceBlockView.tsx`:
- reads `attrs.evidenceId` and `cardCanvasContext.cardId`
- calls `useCardEvidence(cardId)` (returns full list — single fetch shared with the page); filters via `findEvidenceById`
- empty state when id not found OR cardId null
- loaded chrome shows metric_key + metric_value + metric_unit + an evaluation chip ("✓ pass" green / "✗ fail" red / "— pending" amber)
- React.memo on `attrs.evidenceId`

Tests cover: 4 evaluation states + missing id (returns undefined) + null list.

### Task 3 — `GateBlockView` + helpers + tests

`GateBlockView.helpers.ts` exports `formatGateBadge(gate: Gate): { label: string; color: 'green' | 'amber' | 'red' | 'grey' }` and `canSignOffGate(gate, mode): boolean` (always returns true for now; Mode-rule integration ships in 10-06).

`GateBlockView.tsx`:

- when terminal (approved/rejected): show actor + timestamp
- React.memo on `attrs.gateId`

### Task 6 — `AiAssistBlockView` (stub)

`AiAssistBlockView.tsx`:
- reads `attrs.conversationId`
- renders chrome: `✨ AI Assist` + `conversationId: <id-or-blank>` + `Conversational AI for this card. Wiring in a future wave.`
- NO fetch, no React Query, no loading state — just a static stub with the attrs visible

This is intentional. AI Assist has no kernel API yet; shipping a real fetch would 404 every render. The stub gives users a recognizable block they can drop into the doc with `/ai-assist` and have the conversationId persisted; the runtime wiring lights up when the backend exists.
