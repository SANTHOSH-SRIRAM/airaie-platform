<objective>
Demo seed cards in the dev DB currently ship with `intent_spec_id IS NULL`, which is fine for Phase 8's structured page (it shows an "Add Intent" empty state and lets the user create one) but breaks Phase 10's canvas E2E flow: every IntentBlock NodeView empty-states, every ResultBlockView falls through to kind-only renderer pick (no intent-aware path), and the auto-migration writes a near-empty default body. Backfill `intent_spec_id` for every seeded Card so the canvas has real data to render on first open.

Strategy: write idempotent SQL files under `airaie-kernel/scripts/seed/` (one per demo board), each (a) inserts the IntentSpec rows the cards SHOULD point at (one per card.intent_type), (b) updates `cards.intent_spec_id` to point at the matching IntentSpec, and (c) ON CONFLICT DO NOTHING so reruns are safe. Wire the seed file(s) into `dev-start.sh` after the migrations step. No frontend, no kernel-Go changes.
</objective>

<must_haves>
- `airaie-kernel/scripts/seed/board-bracket-3b5179.sql` exists and is idempotent (re-running yields no errors and no row-count drift)
- `airaie-kernel/scripts/dev-start.sh` invokes the seed file(s) after migrations, before runner launch
- Post-`dev-start.sh` cold start: `SELECT COUNT(*) FROM cards WHERE board_id = 'brd_bracket_3b5179' AND intent_spec_id IS NULL;` returns 0
- `SELECT c.id, c.intent_type, i.intent_type FROM cards c JOIN intent_specs i ON i.id = c.intent_spec_id WHERE c.board_id = 'brd_bracket_3b5179';` shows matching intent_type per card pair
- The canvas opens on a seeded card and IntentBlock renders the IntentSpec's goal + intent_type (verifiable once 10-02 lands)
- All other seeded boards (if any — discover via `SELECT DISTINCT board_id FROM cards WHERE id LIKE 'crd_%';`) get the same treatment
- `kernel go build` clean and `dev-start.sh` smoke green (gateway + runner up, /v0/health 200)
</must_haves>

<action>
For each (board_id, card_id, intent_type) from Task 1, generate one INSERT + UPDATE pair. Example for `crd_fea_3b5179`:
```sql
-- IntentSpec for FEA bracket-stress card
INSERT INTO intent_specs (id, board_id, intent_type, goal, inputs, acceptance, status, created_at, updated_at)
VALUES (
  'int_fea_3b5179',
  'brd_bracket_3b5179',
  'sim.fea_stress_analysis',
  'Verify bracket sustains 5kN load with safety factor >= 2.0',
  '[{"name":"geometry","artifact_id":"art_bracket_step_v1"},{"name":"loads","artifact_id":"art_loads_csv_v1"}]'::jsonb,
  '[{"metric_key":"max_von_mises_mpa","operator":"less_than","threshold":210.0}]'::jsonb,
  'locked',
  NOW(),
  NOW()
)
ON CONFLICT (id) DO NOTHING;

UPDATE cards SET intent_spec_id = 'int_fea_3b5179', updated_at = NOW()
WHERE id = 'crd_fea_3b5179' AND intent_spec_id IS NULL;
```
Repeat for every card the audit listed. Use `inputs` `artifact_id` values that already exist in the demo board's artifact pool (verify via `SELECT id, name, type FROM artifacts WHERE board_id = 'brd_bracket_3b5179';` — pick types that match the intent's input ports per the kernel's intent_type registry).

Final shape: one SQL file per board in `airaie-kernel/scripts/seed/`. Idempotent via `ON CONFLICT (id) DO NOTHING` for inserts and a `WHERE intent_spec_id IS NULL` guard on updates.
