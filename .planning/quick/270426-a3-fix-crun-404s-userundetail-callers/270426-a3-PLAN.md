<objective>
The Phase 10 smoke-test session surfaced harmless-but-noisy console errors: `useRunDetail(latestRunId)` is called from three Card-page surfaces (CardExecutionSequence, CardStatusPanel, CardDetailPage) with the latest card-run id. The hook calls `GET /v0/runs/{id}`, which is the workflow-runs path. The kernel returns 404 for `crun_*` ids on that path. The 404 doesn't break behavior (the components have null-tolerant rendering paths) but fills the console with errors that mask real issues during smoke-testing.

Fix: route by id prefix. `crun_*` → kernel's card-run detail endpoint (find via `grep -nE "card-runs|cardrun" airaie-kernel/internal/handler/`); `run_*` → existing `/v0/runs/{id}` workflow path. Either (A) add a new `useCardRunDetail` hook that the three Card surfaces call, or (B) make `useRunDetail` polymorphic on the id prefix. Option A is cleaner (explicit), option B is one-call-site-changed (smaller diff). Pick whichever the codebase already prefers — the existing `listCardRuns` already lives in `api/cards.ts` so option A keeps card-run concerns in `cards.ts` (recommended).

Frontend-only. No kernel changes. The kernel's existing `GET /v0/card-runs/{id}` (or equivalent) is the source of truth — verify it exists and returns the same shape `RunDetail` consumers expect, OR map the response to RunDetail in the frontend adapter.
</objective>

```ts
export function useAnyRunDetail(runId: string | null) {

  const cardRun = useCardRunDetail(isCrun ? runId : null);
  return isCrun ? cardRun : wfRun;
}
```
