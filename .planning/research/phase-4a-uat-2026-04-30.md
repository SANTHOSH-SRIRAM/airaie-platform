## G.4.13 — alert→toast (PARTIAL)

**Test:** Clicked Start Run on `/workflows/wf_96267c6c`. Backend returned 500 (workflow has no executable version) — exactly the failure path the fix is supposed to handle.

**Verified:**
- ✅ `window.alert` is **never called** (hooked alert; `__alertFired` remained `false` across all click attempts)
- ✅ Mutation transitioned to error state (`status: "error"`, `error.message: "Internal server error"`)
- ✅ Per-call `onError` callback fires when invoked directly via `mutationApi.mutate(...)` from a fiber-walked handle
- ✅ `setActionError` is called from the component's `handleStartRun` — confirmed via React fiber inspection: after click, `WorkflowDetailPage.alternate.memoizedState[74]` = `{title: "Could not start run", subtitle: "Internal server error"}`

**NOT verified (failure mode):**
- ❌ `Notification` component does **not** render in the DOM
- `fiber.memoizedState[74]` (current tree) stays `null` — only `fiber.alternate.memoizedState[74]` carries the error object
- Forced direct dispatch via `queue.dispatch({...})` also failed to render
- Module served by Vite has the patched code (verified via `curl http://localhost:3000/src/pages/WorkflowDetailPage.tsx | grep actionError` → matches disk)

**Diagnosis:** The state set is reaching the workInProgress fiber but the commit phase isn't promoting it to `current`. This isolates to either:
1. **Vite Fast Refresh state corruption** — the dev server has been running since 11:17 and HMR'd through ~30 frontend commits including the trio. Adding a new `useState` to a long-lived module via HMR can leave hook indices misaligned between the workInProgress and current fibers.
2. **A genuine concurrent-mode bug** — much less likely; would require a parallel high-priority update starving this one.

**Recommended next step:** Kill `npm run dev` (PID 23962, started 11:17) and restart cleanly, then re-run the Start Run test. If notification renders, it's HMR corruption (no code fix needed). If it still doesn't render, file as new bug — code change was incomplete (e.g. Notification primitive needs a different mounting position).

**Regression risk graded:** LOW for this trio — `window.alert` IS removed (the headline objective), so the worst case is a silent failure where an error has no UI surfacing. Compared to baseline (modal-blocking alert), silent failure is a better UX even before the Notification renders.
